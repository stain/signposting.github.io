API Catalog: Supporting Discovery of a Repository's Interoperability Affordances

Prepared by: Herbert Van de Sompel, Robert Huber, Michael L. Nelson, Martin Klein, Andrea Bollini, ...
Previous version, created 20231126: https://www.signposting.org/API-Catalog/
Previous version, created 20231124: https://www.signposting.org/API-Catalog/
Please provide feedback in the GitHub Signposting repository

In order to meet FAIR expectations, repositories provide affordances to support Finding, Accessing, Interoperating, and Reusing the scholarly objects they manage. Some affordances focus on supporting machine interaction with scholarly content and entail implementing common specifications as well as repository-specific machine interfaces. Discovering whether and which interoperability affordances are provided by a repository remains a challenge. Some specifications describe how to advertise a compliant interface, but, the discovery approach is different for different specifications. Other specifications don't detail an approach to discover a compliant interface and rely on a central registry instead. For yet other specifications, no way to advertise compliance exists. This document describes an approach that repositories can uniformly use to advertise the interoperability affordances they provide.

1. Introduction

This specification details an approach that repositories that manage scholarly objects can use to advertise the interoperability affordances they provide. A repository can use the approach, for example, to:
  • indicate what the URL of its OAI-PMH interface is,
  • point to a Sitemap that lists the objects managed by the repository,
  • convey the URL of a repository-specific API,
  • advertise support for FAIR Signposting,
  • convey that objects managed by the repository can be downloaded as RO-Crates.
The approach consists of:
  • Publishing a single JSON Link Set that contains links to documentation for each available affordance.
  • Making that Link Set discoverable.

Interoperability affordances that can be advertised using the approach can be:
  • Repository-level affordances: Machine interfaces that can be used to interact with the repository as a whole, e.g. the aforementioned OAI-PMH interface.
  • Object-level affordances: Affordances that are available when interacting with a single scholarly object managed by the repository, e.g. the aforementioned support for FAIR Signposting.
The approach leverages the following formal specifications:
  • RF9264: Linkset: Media Types and a Link Relation Type for Link Sets - This RFC defines the application/linkset+json document format that can be used to convey a set of typed links.
  • RFC8631: Link Relation Types for Web Services - This RFC defines the service-doc, service-desc, and service-meta link relation types that can be used to, respectively, link to a human-readable description of a machine interface, a machine-readable description of a machine interface, and additional information pertaining a the machine interface.
  • The api-catalog Internet Draft - This specification defines the api-catalog link relation type that can be used to point at a machine-readable document that provides an overview of the machine interfaces for a repository. The draft has been adopted by the IETF HTTP API Working Group and will eventually be published as an RFC.

2. Building Blocks

This section provides a description of the building blocks of the API Catalog approach that can be used to provide details about the interoperability affordances that are supported by a repository.

2.1. The API Catalog Link Set

The API Catalog is a Link Set that is serialized in the application/linkset+json format. The content of the Link Set is as follows:
  • For each interoperability affordance provided by a repository, the Link Set contains a link context object with a mandatory anchor member. The URL provided as value of the anchor member depends on the nature of the affordance:
    • Repository-level affordance: The value of the anchor member is the URL of the machine interface. If more than one affordance of the same type exist, a distinct anchor member must be used for each.
    • Object-level affordances: The value of the anchor member is the URL of the landing page of a single example object managed by the repository that illustrates the affordance. If more than one object-level affordance exists, each must be illustrated by means of a different example object. Only object-level affordances that are consistently available for each object managed by the repository must be considered.
  • The anchor member is the context for links to resources that provide information about the interoperability affordance. Links with the service-doc, service-desc, and service-meta link relation types can be provided; the semantics of these relation types are described in Section 2.2. Links with other link relation types must not be provided.
  • For each link, the type link target attribute must be used to convey the media type of the resource that is the target of the link. The profile attribute may be used to provide further specificity regarding the media type, and, if used, must contain a profile URI. The title attribute may be used to improve human readability of the Link Set, and, if used, must contain the title of the resource that is the target of the link.
  • All URLs in the Link Set must be absolute URIs; relative URIs must not be used.

2.2. Link Relation Types to Point at Descriptions of Interoperability Affordances

RFC8631 defines the following link relation types that are used in an API Catalog Link Set to point at various descriptions of a specific interoperability affordance that is provided by a repository:
  • service-doc: This link relation type is used to point at a human-readable specification of an interoperability affordance. It can, for example, be used to point at the URI of a specification or standard, e.g. the OAI-PMH specification, the SPARQL 1.1 W3C Recommendation, the FAIR Signposting specification.
  • service-desc: This link relation type is used to point at a machine-readable specification of the interoperability affordance. It can, for example, be used to point at the URI of an OpenAPI or RESTdesc specification of an API exposed by the repository.
  • service-meta: This link relation type is used to point at further details regarding the interoperability affordance. It can, for example, be used to point at the Identify endpoint of an OAI-PMH interface or a SPARQL 1.1 Service Description document.

2.3. Discovery of the API Catalog Link Set

A repository's API Catalog can be discovered via the root-URL of a repository. The root-URL is the top of the repository's path's hierarchy. For example:
  • For a repository at URL <https://myuniversity.edu/repository/home> the root-URL is <https://myuniversity.edu>.
  • For a repository at URL <https://repository.myuniversity.edu/home> the root-URL is <https://repository.myuniversity.edu>.

Two discovery approaches leverage the root-URL, the /.well-known/api-catalog and the api-catalog link, detailed in Sections 2.3.1. and 2.3.2., respectively. These approaches should be used simultaneously and least one must be used. Moreover, the repository's API Catalog can also be made discoverable by means of api-catalog links at the landing pages of objects managed by the repository; this is detailed in Section 2.3.3.
2.3.1. /.well-known/api-catalog at the root-URL
The behavior for a repository's /.well-known/api-catalog URI should be as follows:
  • When a repository with root-URL receives an HTTP GET request against root-URL/.well-known/api-catalog, the request must resolve to its the API Catalog Link Set serialized in the application/linkset+json format.
  • When a repository with root-URL receives an HTTP HEAD request against root-URL/.well-known/api-catalog, the request must resolve to a response that contains an HTTP Link header with a link that uses the api-catalog link relation type to point at the repository's API Catalog Link Set. Further details about this link are provided in Section 2.3.2.
2.3.2. api-catalog link at the root-URL
In order to support discovery of its API Catalog Link Set, a repository should provide a link from its root-URL to the API Catalog Link Set with the api-catalog link relation type. This link:
  • Must be provided in the HTTP Link header and/or the HTML <link> element. If provided in the HTTP Link header, both HTTP HEAD and HTTP GET requests must provide the api-catalog link.
  • Must have a type attribute with value application/linkset+json.
  • Must have a profile attribute with value https://signposting.org/API-Catalog/.
2.3.3. api-catalog link at landing pages
Repositories should consider to also provide an api-catalog link pointing from landing pages of objects managed by the repository to the API Catalog Link Set. Such a link:
  • Must be provided in the HTTP Link header and/or the HTML <link> element. If provided in the HTTP Link header, both HTTP HEAD and HTTP GET requests must provide the api-catalog link.
  • Must have a type attribute with value application/linkset+json.
  • Must have a profile attribute with value https://signposting.org/API-Catalog/.

3. Examples

3.1. Discovery

This section illustrates discovery of an API Catalog using the well-know URI and api-catalog link approaches.
3.1.1. Discovery via /.well-known/api-catalog at the root-URL
The below example illustrates how an agent can discover a repository's API Catalog using the well-know URI approach. It shows the response to an HTTP HEAD request issued against the /.well-known/api-catalog URI at the root-URL <https://my.repo.org/> of the fictitious repository <https://my.repo.org/home/>. The response contains an HTTP Link header that has a link with the api-catalog link relation type and the mandatory type and profile attributes as described in Section 2.3. The link's target <https://my.repo.org/api-info.json> is the URL of the repository's API Catalog Link Set. The client can proceed by issuing an HTTP GET against that URI to obtain the API Catalog.

HTTP HEAD against a repository's /.well-known/api-catalog URI: request and response 
$ curl -I "https://my.repo.org/.well-known/api-catalog"

HTTP/1.1 200 OK
Date: Sat, 18 Nov 2023 14:43:06 GMT
Link: <https://my.repo.org/api-info.json> 
      ; rel="api-catalog" 
      ; type="application/linkset+json" 
      ; profile="https://signposting.org/API-Catalog/" 
   Content-Length: 0
   Connection: close
3.1.2. Discovery via an api-catalog link
The previous example shows how a client discovers an api-catalog link by interacting with the /.well-known/api-catalog URI and can follow that link to retrieve the API Catalog. But api-catalog links can also be provided at the level of landing pages of objects managed by a repository.

The below example shows an agent issuing an HTTP HEAD request against the landing page URL <https://my.repo.org/item/747369/> of an object managed by the fictitious repository <https://my.repo.org/home/>. The response contains an HTTP Link header that has a link with the api-catalog link relation type and the mandatory type and profile attributes as described in Section 2.3. The link's target <https://my.repo.org/api-info.json> is the URL of the repository's API Catalog Link Set. For brevity, FAIR Signposting links that should be provided in the same response are not shown.

HTTP HEAD against the landing page URL of an object managed by a repository: request and response 
$ curl -I "https://my.repo.org/item/747369/"

HTTP/1.1 200 OK
Date: Sat, 18 Nov 2023 14:43:12 GMT
Content-Type: text/html
Content-Length: 2657
Link: <https://my.repo.org/api-info.json> 
      ; rel="api-catalog" 
      ; type="application/linkset+json" 
      ; profile="https://signposting.org/API-Catalog/"

3.2. API Catalog Link Set

This section illustrates API Catalogs that have repository-level and object-level entries, respectively. In actual API Catalogs, these entries can be combined. But, here, they are shown separately for increased clarity.
3.2.1. Repository-Level Affordances
The below example shows an API Catalog listing only repository-level interoperability affordances. For such affordances, the value of the anchor member is the URL where the affordance is accessible:
  • The affordance at <https://my.repo.org/oaipmh> is detailed in a human-readable manner (service-doc link relation type) by the document at <https://www.openarchives.org/OAI/openarchivesprotocol.html>, which is the OAI-PMH specification. Further information about the repository's OAI-PMH implementation is provided by means of a link with the service-meta link relation type that points at the URL for the Identify request against the OAI-PMH interface.
  • The affordance at <https://my.repo.org/sitemap/index.xml> is detailed in a human-readable manner (service-doc link relation type) by the document at <https://www.sitemaps.org/protocol.html>, which is the Sitemap protocol specification. No further information is provided for this affordance.
  • The affordance at <https://my.repo.org/sparql11> is detailed in a human-readable manner (service-doc link relation type) by the document at <https://www.w3.org/TR/sparql11-query/>, which is the SPARQL 1.1 W3C Recommendation. Further information about the repository's SPARQL end-point is provided by means of a link with the service-meta link relation type that points at the URL of a SPARQL Description document.
  • The affordance at <https://my.repo.org/api> is detailed in a machine-readable manner (service-desc link relation type) by the JSON document at <https://my.repo.org/api/api-specification.json>, which could be an OpenAPI document. It is also detailed in a human-readable manner (service-doc link relation type) by the document at <https://my.repo.org/api/api-specification.html>.

3.2.2. Object-Level Affordances
The below example shows an API Catalog listing only object-level interoperability affordances. For such affordances, the value of the anchor member is the URL of the landing page of a single example object managed by the repository that illustrates the affordance. Only object-level affordances that are consistently available for each object managed by the repository must be listed in the API Catalog.
